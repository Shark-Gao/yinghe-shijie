# 电视剧解说短视频工坊

将有权使用的电视剧、电影或短剧素材，剪成以中文剧情解说为主体的短视频。复用 `yinghe-short-video` 的审片、计划、配音和导出能力，但不沿用硬核科普的选题与叙事方式。

## 视频号选题方向

将视频号作为默认发布场景时，优先做**年代家庭伦理剧解说**。围绕家庭、婚姻、赡养、兄弟姐妹、返乡、婆媳、遗产、重组家庭和女性成长，选择观众能马上代入、也适合转发讨论的一个具体矛盾。

候选剧情按以下优先级筛选：

1. 一家人因房子、赡养、婚姻或责任分配被迫作出选择，且本条素材中有明确结果。
2. 八九十年代或跨年代生活变迁中的亲情、手足关系、返乡与个人成长。
3. 女性从忍让到作出决定，或重组家庭重新建立边界；不得把羞辱、性别对立或苦难消费当作钩子。
4. 单集或一段素材已形成“误会/危机 → 升级 → 真相或选择”的完整冲突链。

第二测试方向为**古装神话、民间传奇与历史传奇**。优先因果、孝道、命运选择、民间人物和局势反转；不要只依赖“神仙打架”“大结局”或角色名做标题。

默认不优先纯甜宠、都市偶像职场和人物关系过多的权谋线，除非该段能压缩为一个清楚的家庭/命运冲突并通过评分门槛。

标题和开头从具体问题切入，不以“第几集发生了什么”或“第 X 集/大结局”作为唯一钩子。例如：

```text
儿子把母亲接进城，本以为是享福，第一顿饭却让这个家变了味。
三个女儿争的不是房子，而是谁愿意真正照顾父亲。
她忍了二十年，直到丈夫把家产交给了外人。
```

这些只是结构示例。只能在素材、字幕和可见画面证实后使用，不得把示例当成真实剧情或虚构冲突。

## 人性问题叙事规则

不得把短视频做成“第几集发生了什么”的剧情流水账。每条计划必须先写出一个能被本条素材回答的**人性问题**，并让标题、前 8 秒口播和结尾共同回答它；集数、剧名和角色名只用于补充检索与背景，不得充当唯一卖点。

可从亲情责任、婚姻边界、手足公平、命运选择、个人成长、善恶代价中选择切口。例如“父母老了，到底该谁负责？”“忍让是在维系家庭，还是在纵容伤害？”“明知会付出代价，还要不要坚持？”。问题必须对应本条可见剧情，不能为了煽情夸大或编造。

在“剧情卡”中新增并先完成这一行：

```text
本条人性问题：<一个由本条剧情回答的具体选择、关系或代价>
```

如果无法写出清楚的人性问题，或素材只能复述剧情进度，则不选该段；改选冲突和结果更完整的素材。

## 边界与输入

1. 以平台影视项目实际展示的可用片单和当前授权状态为唯一依据。创作前确认作品仍有效、未临期；后续定期查看片单和平台通知。不得下载或绕过付费、登录、DRM、平台权限；不移除版权标识，也不声称剧集归自己所有。
2. 只使用当时可免费或可供会员观看的内容；不得使用额外付费内容，包括“超前点播”“云首发”等。
3. 成片目的仅限介绍、宣传、讨论相关影视项目的自行使用；不要将素材用于与该项目无关的叙事、商品导流或冒充官方内容。
4. 不得形成可能实质替代正片传播的效果：禁止搬运、拆条，禁止用一条或少量成片将剧情从头到尾讲完。每条只讲一个可独立成立的冲突链，并保留观看正片所需的关键过程与整体体验。
5. 遵守单条成片时长上限：剧集、综艺不超过 4 分钟；电影不超过 3 分钟；微短剧不超过 2 分钟；动画、纪录片不超过 90 秒。用户指定的时长超过上限时，拒绝按该长度导出并改为合规时长；如平台另有比例或其他要求，以其完整授权规则为准。
6. 需要原始视频与对应字幕（SRT、VTT 或可核对的台词文本）。只有截图、链接或剧情梗概时，只交付解说文案和剪辑计划，不虚构时间码或成片。
7. 用中文口播、重组镜头和明确的剧情观点，使成片具有独立解说价值；不要只顺序拼接高光、逐句翻译台词或复述整集。
8. 除非用户明确指定，不使用原剧人声；`source_volume` 设为 `0.0`。对白确有叙事必要时，只保留短促、可核验的原声，并在解说停顿处使用。

## 制作流程

1. 运行 `../yinghe-short-video/scripts/prepare_review_assets.py`，检查时长、字幕提示和联系表。只根据用户提供的本地素材选镜头。
2. 先建立一张“剧情卡”，再选镜头：

   ```text
   本条核心冲突：<谁为了什么，陷入什么困境>
   本条人性问题：<一个由本条剧情回答的具体选择、关系或代价>
   观众最早应知道的结果：<反转 / 代价 / 关键选择>
   人物关系：<角色名或身份 + 与主角的关系>
   起因 → 升级 → 转折 → 当集落点：<各一句>
   不能断言的内容：<未在素材或字幕中证实的动机、后续剧情>
   ```

   人物多、剧集长时，只讲一个冲突链；不要把全剧人物和支线塞进一条短视频。
3. 为每个候选剧情链评分：冲突强度 25%、反转/结果 25%、人物动机清晰 20%、镜头可证明 15%、脱离全集仍能看懂 10%、9:16 可读性 5%。优先选择符合“视频号选题方向”的高分故事；导出所有高分且不重叠的故事，每条单独建计划和输出目录。
4. 默认做 45–75 秒、9:16 的剧情解说，并在计划中设 `layout: "contain_blur"`。若主体位置清楚、裁切后仍保留人物关系和表情，才用 `fill_crop`；否则保持 `contain_blur`。无论平台版本如何调整，均不得超过“边界与输入”中的对应作品单条时长上限；B 站长版仅在用户明确要求且仍合规时另做。
5. 用“结果先到，原因随后”的顺序写中文口播：

   - 0–2 秒：强冲突或结果，例如“他以为自己被救了，下一秒却发现全是陷阱。”
   - 2–8 秒：说清人物、关系和当前危机。
   - 中段：每 3–7 秒推进一次起因、选择、阻碍或新信息；每一句都要有对应画面。
   - 后段：交付本条承诺的反转、代价或阶段性结果；不把“下一集见”当作唯一结尾。
   - 结尾：给出贴合剧情的问题或下一条的具体悬念，不加泛化的“点赞关注”。

   用角色名或清晰身份指代人物。不要编造心理活动、关系、结局或台词；不把未播出的后续当作已发生事实。避免“震惊”“太狠了”“全网泪目”等空泛词。
6. 在写计划前，按 `../yinghe-short-video/references/edit-plan-schema.md` 建立 JSON，并额外加入以下仅供编辑审核的字段：

   ```json
   {
     "drama": {
       "work_title": "剧名",
       "episode": "第119集",
       "story_arc": "主角被迫加入对手阵营",
       "spoiler_scope": "仅揭示本条素材内的阶段性结果",
       "rights_note": "用户确认有权将该素材用于本次制作"
     }
   }
   ```

   保持原计划中的 `platform_titles`、`platform_descriptions`、`clips`、`narration` 和 `mix` 字段。所有剪点必须落在完整口播段结束处；不得在一句话中途切换到不相关人物或时间线。
7. 标题、封面和发布简介必须都承诺同一个剧情结果，并围绕已写明的“本条人性问题”。标题突出身份反差、选择代价、隐藏关系、局势逆转或具体结果；不要用“第119集”“大结局”作为唯一钩子。小红书标题可加剧名和人物关系便于搜索，抖音/快手则更口语直接。
8. 默认使用 `zh-CN-XiaoxiaoNeural`、`+0%`，语气是清晰克制的讲故事，不演角色、不夸张喊叫。音乐仅可使用项目中确认可用的曲目，并在计划和发布简介保留必要署名；没有适合的授权 BGM 时，先生成纯配音，不擅自挪用音乐。
9. 调用 `../yinghe-short-video/scripts/produce_short_video.py --plan <plan.json>` 导出。首句前留 0.3–0.6 秒视觉起势，最后一句后留 0.5–0.8 秒收束；普通口播段间保持约 0.2–0.4 秒，不留下无意义的空白。
10. 逐帧抽检并完整试听。确认：剧情在前 8 秒可懂、人物与字幕对得上、反转有画面证据、竖屏没有裁掉关键表情/人物关系、配音和 SRT 同步、标题不剧透超过本条素材。通过后再运行 `../yinghe-short-video/scripts/cleanup_export.py` 清理中间文件。

## 封面规则

只在用户明确要求封面时制作。优先用不含演员肖像的剧情化视觉设计（关系符号、场景、道具、色块与文字），或用户确认可用的官方物料；不要生成可识别演员的仿真人脸、伪造剧照或误导性剧情画面。文字后期叠加，底图不依赖生成模型写中文。

## 交付

每条成片交付：最终 MP4、SRT、计划 JSON、四个平台标题和发布简介，以及在已要求时的三种封面。简要说明剧名/集数、核心冲突、时长、作品类别、配音、是否保留原声、当前片单/授权确认结果及适用的时长上限。
